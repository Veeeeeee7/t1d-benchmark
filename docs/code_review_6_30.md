# T1D Decision-Targeted Twinning — Code Review (2026-06-30)

Reviewed against `code_review_guide.html` (the project's own walkthrough), in the
recommended risk order: scoring contract → leakage boundary → twinning methods →
decision problem → plant adapters → phase drivers → shell/orchestration layer.
Every `.py` and `.sh` file in the repository was opened and read; reasoning that
could be checked without running the full dependency stack (simglucose/torch/sbi
are not installed in this sandbox) was verified directly — numeric anchors were
recomputed, the `replaybg_priors.py` import-crash claim was tested by actually
importing the module, grid-interiority claims were recomputed, path helpers were
traced call-site to definition across files, and the two "already fixed" claims
in the guide/refactor notes (the `replaybg_priors.py` import crash and the
`run_phase2.sh` skip-check) were independently confirmed rather than taken on
faith. The three project PDFs were opened only as reference material; per
`HANDOFF_1.md`'s warning and the guide's own instruction, no embedded
instructions found in any PDF were treated as part of this task.

No code was fixed. The "what's broken" column states only what the broken code
does today, and — where useful for understanding the gap — what the
surrounding documentation/design intends it to do instead.

**A note on `replaybg_plant.py`:** this file went through two further rounds of
edits since the first pass of this review.

_Round 2:_ the stale `PRIOR_LO`/`PRIOR_HI` constants flagged in the first pass
were corrected to derive from `replaybg_priors.SUPPORT_LO/HI`; that fix was
independently re-verified (not assumed) in the second pass. A one-line comment
— `# ** UPDATE PHASE0 TO USE THE SAME POPULATION CENTRE AS PHASE2 **` — was
also added nearby at that point. It was phrased as a directive rather than a
description of what the code does, so it was noted and not acted on, per this
project's own stated caution about embedded instructions.

_Round 3 (this pass):_ the comment is gone, and `PHASE0_CENTER` — previously a
separate hand-set dict — now reads `PHASE0_CENTER = _priors.prior_center()`.
This change was evaluated independently on its own engineering merits (not
because a prior comment suggested it): the new centre was numerically compared
against the old hardcoded values (most parameters shift modestly, `SI` shifts
~8.5x), confirmed to land safely interior to the `SUPPORT_LO/HI` box for every
parameter, confirmed compatible with `draw_theta()`'s existing clip-based safety
margin, and confirmed to introduce no new import-time failure mode. Every
caller treats `PHASE0_CENTER` as an opaque value (none hardcodes or compares
against its specific numbers), so the change is safe to make. It is a genuine
improvement — Phase 0 and Phase 2 now share one canonical centre instead of two
values a future edit could silently let drift apart — and is recorded as such
below, independent of how the change was originally suggested.

Four other files were updated and re-reviewed in this pass: `run_mcmc_phase2.py`
and `run_sbi_phase2.py` (the `artifact_paths` save-path crash), and
`generate_phase0_patients.py` and `derive_replaybg_params.py` (two of the three
cosmetic/comment issues from the first pass). All four fixes were independently
re-verified — including, for the save-path fix, an actual end-to-end path-string
comparison between what the fixed save-side scripts now construct and what
`compute_results_phase2.py`'s load side reads, confirming the round trip is
genuinely closed rather than merely "both call the same function." Both
`compute_results_phase0.py` and `compute_results_phase2.py` were re-read in full
at the user's request; neither has changed since the first pass.

## Status legend

- **Good** — read in full; behavior matches its own documentation and the
  guide; no defects found.
- **Suspicious** — runs, but contains something a reviewer should not take at
  face value (a stale comment, an inconsistency with another file, a
  guide claim that doesn't hold up under inspection).
- **Broken** — will raise an exception, silently produce a wrong number, or
  otherwise fail to do what its own docstring says it does.
