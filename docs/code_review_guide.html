1   What you are reviewing

The codebase is a research benchmark. It asks a single scientific question: when you build a simplified “digital twin” of a diabetes patient from their glucose data, should you judge that twin by how closely its glucose trajectory matches reality (fidelity), or by whether it picks the same best therapy the real patient would (decision transfer)? The thesis is that these come apart — a twin can track the data well yet rank therapies wrong, and vice versa — and the whole pipeline is engineered to measure that gap. The authors call this DT² (decision-targeted twinning).

The Two Models — the load-bearing idea

Everything depends on a deliberate mismatch between two glucose models:

  • The plant (ground truth) is simglucose (the UVA/Padova simulator) — a complex ~13-state model treated as the “real patient.” It is a black box: you only see the CGM trace it emits.
  • The twin family is ReplayBG — a simplified 8-parameter ODE. It cannot reproduce the plant exactly. The authors write this mismatch as “Φ not in F” and it is the entire point: a perfect twin is impossible by construction, so the question of which imperfection matters becomes meaningful.

A “digital twin” is therefore nothing more than a vector of 8 ReplayBG parameters. A “twinning method” is a recipe for turning one CGM trace into those 8 numbers. Three such methods are compared (next section).

Caveat — the core invariant, the easiest thing to accidentally break
  • Every method maps CGM → ReplayBG parameters, and every method’s machinery stays inside the ReplayBG family. A method must never train on simglucose simulations — that would leak the true dynamics the other methods can’t see (“simulator-family leakage”) and would break fairness and contaminate the “Φ not in F” premise.
  • The only simglucose data any method may use is the single CGM trace it is twinning.
  • What changed: patient information no longer enters through a cohort-derived prior. The shared prior is now the published ReplayBG prior (replaybg_priors.py), which carries zero cohort/simglucose information — so the fairness story is actually cleaner. The per-patient label (rbg_theta) is a MAP projection of that patient’s own CGM under the published prior: still a ReplayBG-coordinate projection that discards the “Φ not in F” residual. Your first question reviewing any new path is still: does in-class (ReplayBG-only) training hold?

How a result is produced (the pipeline in one breath)

For one patient: (1) run a baseline therapy on simglucose to get an identification CGM trace; (2) each method turns that trace into ReplayBG parameters (a twin); (3) replay a grid of candidate therapies on both the real plant and the twin; (4) score how well the twin’s therapy ranking matches the plant’s true ranking. Repeat over ~1000 synthetic patients and aggregate.

2   Read this before you open a single file

Four things will mislead you if you don’t know them going in. None is a bug — they are facts about the state of the delivery.

Caveat — the real project is two packages

The code imports as from t1d_twin import … and from experiments import …, with relative imports. It expects a t1d_twin/ core package and an experiments/ suite package at the repo root; review against that layout, not a flat dump. The new replaybg_priors.py lives in t1d_twin/ (core).

  • HANDOFF_1/2/3.md are point-in-time snapshots and predate several large changes; treat them as historical.
  • In-code docstrings were brought in sync on 6/27 across ~14 modules (the MAP/published-prior refactor and the 168 h→24 h horizon), so module/function docstrings in the changed areas are now reliable. The markdown handoffs are still stale — prefer the code and the dated notes.
  • Current dated notes, in order: carb_error_perturbation_6_17.md, logistic_regression_and_refactor_6_17.md, phase_0_implementation.md, population_prior_refactor_6_26.md (empirical-Bayes prior — now historical), MAP_published_prior_refactor_6_27.md (current state of the prior/fit layer; supersedes the 6/26 note). Axis 1 — the three twinning methods (what gets compared)

All three emit 8 ReplayBG numbers and run through the same replay/scoring code. PointTwin subclasses SBITwin with a single-row posterior (degenerate uncertainty by design). All three now share the informative published prior via replaybg_priors — MCMC’s prior is no longer uniform, SBI’s is no longer BoxUniform, and the per-patient label fit (fit_replaybg) is MAP under that same prior, so one citable object is threaded through fit, MCMC, and SBI.

Axis 2 — the three phases (which plant / labels)

phaseplantmethodsanswers
Phase 0ReplayBG (matched)MCMC, SBI, TCN, ridgeBest case: how well do methods rank when the model is correctly specified? The ceiling.

Review checks — why Phase 0 exists

Under the mismatch, MCMC was seen to collapse to ~0 rank correlation and SBI showed huge regret outliers — which looked like a harness bug. Phase 0 makes ReplayBG its own plant and shows the same code scores near-perfectly (rank corr ~0.95–1.0). That dissociates “the methods are broken” from “the mismatch is hard.” Confirm this dissociation reproduces — it is the spine of the project’s defence. (Phase 1 labels are now MAP projections, so they are mildly prior-informed; this slightly blurs Phase 1 as a “pure data-only” baseline — see §6.)

6   Consolidated verification checklist

Passing tests do not mean the result is sound — tolerances are loose by design.

Review checks — correctness of the scoring core  (reviewed; unchanged)
  • Reproduce the value.py anchors by hand (112.5 → reward ~0; sustained lows → high LBGI/TBR; highs → high HBGI/TAR; TIR+TBR+TAR ≈ 100%).
  • Re-derive the unit conversions independently; spot-check one run’s insulin_mU_kg_min / CHO_mg_kg_min.
  • Confirm metrics are computed on noise-free IG on both sides, and RunResult.bg() is the noise-free signal.
Review checks — fairness / no leakage (updated for the published prior)
  • All three methods receive the same published prior (replaybg_priors), and no method trains on simglucose sims.
  • Confirm the shared prior carries no cohort information — it is literature constants. (The old empirical-Bayes / leave-one-out concern is therefore moot.)
  • group_kfold_indices: no patient in both train and test; every patient tested once.
  • Phase 1 labels are ReplayBG MAP projections (carry proj_rmse); Phase 0 labels are exact (proj_rmse = 0).
  • The old “lifted 2e-3” SI box bound no longer applies — SI now follows the Gamma(3.3, 5e-4) prior; the support envelope tops out at ~4e-3. Confirm the support box never censors a fit.
  • Ordering enforced: verify ka2<kd and kabs<kempt hold for every MAP label (a RuntimeWarning fires otherwise) and that MCMC/SBI reject violations.
Review checks — MAP labels & the published prior (new)
  • p2 is pinned by its prior and is therefore a near-constant, trivial regression target. Characterise per parameter which labels are likelihood-driven (SI, Gb, kabs) vs prior-driven (p2, partly SG); don’t credit a regressor for “predicting” a constant.
  • Shrinkage / centre: MAP pulls each fit toward the published (real-patient) centre, while the cohort is UVA/Padova projections. Overlay the MAP-fit histograms against the published densities and report how much the prior moved them — this is the trust argument and where a foreign centre would show up.
  • replaybg_model.VI = 0.126 vs upstream 0.135: the priors were calibrated against upstream’s VI; this shifts steady-state Ipb = Ib/(VI·ke) by ~7%. Note it, or align VI for strict consistency with the prior’s source. (Flagged here as a prior-calibration property, not a change to the reviewed model file.)
  • Phase 1 is now mildly prior-informed (labels are MAP, not pure LS), so it is no longer a clean “data-only” contrast to the Bayesian twins. Fine for a benchmark, but state it.
  • Re-derive before trusting anything: old LS rbg_* columns are stale. Run derive_replaybg_params → rebuild Phase 1 dataset → re-run phases.
Review checks — a non-trivial, well-posed decision problem
  • For each subject, the true optimum over the grid is interior, not an endpoint (bolus_x0.70 or x2.00).
  • bolus_x1.00 is rarely the per-patient best (the carb error did its job).
  • No baseline run truncates into the worst-case padding; per-patient reward span clears the noise floor (inspect_reward_separation.py).
Review checks — the headline scientific claim actually fires
  • Phase 0 (matched) scores near-ceiling while Phase 2 (mismatch) degrades — the designed gap, not a bug.
  • Look for the dissociation: a twin with good trajectory RMSE but poor ranking (or vice versa). If fidelity and ranking move in lockstep, the experiment isn’t exercising the DT² thesis.
  • Each twin actually tracks the identification CGM — open the fit figures; don’t trust RMSE alone. Expect Gb, SI, and now p2 to be well-pinned (the first two by data, p2 by its prior); the rest stay non-identifiable.
Review checks — reproducibility & ops
  • Seeds fixed everywhere; a given config reproduces its table exactly.
  • Thread pinning (OMP_NUM_THREADS=1 etc.) is set before forking the per-patient pools.
  • Clear stale 168 h artifacts before re-running at the 24 h window. Also clear stale LS rbg_* columns / population .npz artifacts before the first MAP run.

Reward-sum(risk) over the horizon; maximised by a flat 112.5 mg/dL trace. The ranking objective.
Spearman / regretRank correlation of twin-vs-true therapy ordering; true reward lost by following the twin’s top pick.
Carb-counting error (m)A deliberate per-patient mis-set of the programmed carb ratio so the optimal corrective bolus f* ≈ m is interior — makes ranking non-trivial.
seen / unseenTherapies near vs far from the baseline the twin was identified on; the optimum sits in “unseen” (a real extrapolation test).
Phase 0 / 1 / 2Matched-model ceiling (ReplayBG plant) / amortised regressors / per-instance MCMC & SBI under the real mismatch.
MCMC / SBI / TCN·ridgePer-instance Bayesian posterior / amortised neural posterior / amortised point estimate of θ.
Published priorThe shared, leakage-free ReplayBG prior taken directly from Cappon et al. (2023) (in replaybg_priors.py): LogNormal rates, Gamma SI, TruncNormal Gb, √-Normal p2, with ka2<kd / kabs<kempt ordering gates. Replaces the former cohort-derived empirical-Bayes prior.
MAP fitThe per-patient label fit (fit_replaybg): penalised least-squares = maximum-a-posteriori θ under the Gaussian CGM likelihood and the published prior. Replaces the former plain least-squares fit.
Support envelopeSUPPORT_LO/HI — a generous physical box that only bounds the optimiser / clamps MCMC walkers. Not a prior; the informative prior does the regularising.
